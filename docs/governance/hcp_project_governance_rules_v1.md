# Human Capability Project (THCP)
# Project Governance Rules V1

Status: LOCKED

## 1. Artifact Authority

Artifacts are the authoritative record of the Human Capability Project. Decisions, rules, charters, schemas, protocols, and implementation guidance become authoritative only when captured in maintained project artifacts.

Conversations are process. They may produce, refine, or challenge decisions, but they do not hold final authority.

Authority never resides in chats.

## 2. Active Projects

Active projects contain current authority. Each active project holds the live artifacts for its subsystem, including current governance, design, schemas, examples, tests, tools, and documentation.

When there is a conflict between an active artifact and a conversation, the active artifact governs.

## 3. Archive Projects

Archive projects contain historical process. They preserve prior conversations, superseded reasoning, obsolete drafts, and context that may explain how decisions were reached.

Archive projects do not define current authority unless an active artifact explicitly restores or incorporates archived material.

## 4. One Active Chat Rule

Each subsystem may have one active chat at a time. The active chat is used for current process work on that subsystem only.

The active chat is not an authority store. Its purpose is to generate and update authoritative artifacts.

## 5. Chat Rotation

When a chat is complete, stale, overloaded, or superseded, it must be rotated out of active use. Archived chats are moved to the matching archive project for the subsystem they concern.

A new active chat may begin only after the current authoritative state is represented in artifacts or in an explicit authoritative hand-off.

## 6. Authoritative Hand-Offs

Hand-offs must identify the current active artifacts and any pending decisions that still require artifact capture.

A hand-off may summarize process history, but it must not make chat history authoritative. Any rule, decision, or requirement needed by future work must be placed in an artifact.

## 7. Subsystem Ownership

Subsystems own their active artifacts. Each subsystem is responsible for keeping its governance, charters, protocols, examples, schemas, tools, and tests aligned with current project authority.

Cross-subsystem decisions must be reflected in every affected active project artifact.

## 8. Scalability

The project structure must scale by separating current authority from historical process. Active projects remain compact, navigable, and current. Archive projects absorb historical chat material and superseded context.

This structure allows many subsystems to evolve without making chat logs the operating memory of the project.

## 9. Core Maxim

Conversations are process.

Artifacts are authority.

Active projects contain current authority.

Archive projects contain historical process.

Authority never resides in chats.
